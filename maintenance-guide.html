my-knowledge-base 网页维护与更新说明

本文档用于指导维护人员更新 Astro + Starlight 知识库站点，包括本地运行、内容编辑、导航配置、资源管理、构建检查和发布前核对。

项目目录：my-knowledge-base 技术栈：Astro 7 + Starlight 主要内容：Markdown / MDX

1. 项目概览

my-knowledge-base 是一个基于 Astro 和 Starlight 的静态文档站，适合维护学习笔记、参考资料、教程和知识索引。 页面内容主要由 src/content/docs/ 下的 .md 与 .mdx 文件生成。

路径 用途
astro.config.mjs 站点地址、基础路径、Starlight 标题、社交链接、侧边栏配置。
src/content/docs/ 知识库正文内容。新增 Markdown 或 MDX 文件后会生成对应页面。
src/content/docs/guides/ 学习笔记、教程、步骤说明。
src/content/docs/reference/ 参考资料、概念说明、速查内容。
src/assets/ 由页面引用并参与构建处理的图片资源。
public/ 不经过处理、原样复制到站点根路径的静态文件。

2. 本地环境

所有命令默认在 my-knowledge-base 子目录内执行。

cd my-knowledge-base
npm install
npm run dev

常用脚本如下：

命令 说明
npm run dev 启动本地开发服务器，通常访问 http://localhost:4321。
npm run build 执行生产构建，输出目录为 dist/。
npm run preview 本地预览已构建的生产版本。
npm run astro 调用 Astro CLI，适合执行框架相关命令。
首次维护时先执行 npm install。如果 node_modules/ 已存在但依赖异常，可重新安装依赖。

3. 内容更新

新增页面

  1. 根据内容类型选择目录：教程放入 guides/，参考资料放入 reference/。
  2. 创建 .md 或 .mdx 文件，文件名建议使用英文小写和短横线，例如 python-notes.md。
  3. 在文件开头添加 frontmatter。
  4. 保存后运行开发服务器检查页面是否出现在侧边栏中。
---
title: 页面标题
description: 页面简介，用一句话说明本文内容
---

## 一级小节

正文内容。

修改现有页面

  • 直接编辑 src/content/docs/ 下对应的 Markdown 或 MDX 文件。
  • 标题变化时同步检查站内链接和侧边栏显示。
  • 删除页面前先确认没有其他页面链接到该页面。

写作规范

  • 主要内容使用中文，代码、命令、文件名和专有名词保持原文。
  • 每篇文档建议包含清晰的标题、简短说明和分级小节。
  • 命令和路径使用代码格式，例如 npm run build。
  • 长篇内容按主题拆分成多个页面，避免单页过长。

4. 导航与站点配置

站点标题、GitHub 链接、部署基础路径和侧边栏配置位于 astro.config.mjs。 当前侧边栏会自动读取 guides 与 reference 目录。 

sidebar: [
  {
    label: '学习笔记',
    items: [{ autogenerate: { directory: 'guides' } }],
  },
  {
    label: '参考资料',
    items: [{ autogenerate: { directory: 'reference' } }],
  },
]

如果新增新的内容分类，例如 projects/，需要：

  1. 在 src/content/docs/ 下创建新目录。
  2. 在 astro.config.mjs 的 sidebar 中增加对应分组。
  3. 运行 npm run build 检查配置是否有效。
当前站点设置了 base: '/my-knowledge-base'。部署到 GitHub Pages 项目页时不要随意删除该配置，否则线上资源路径可能失效。

5. 图片与静态资源

页面图片

推荐将页面内引用的图片放入 src/assets/，然后在 Markdown 或 MDX 中使用相对路径引用。 这种方式便于 Astro 在构建时处理图片资源。 

![图片说明](../../assets/example.webp)

公共静态文件

public/ 中的文件会原样复制到最终站点，适合放置 favicon.svg、下载文件或无需构建处理的资源。

资源命名建议

  • 文件名使用英文小写、数字和短横线。
  • 图片优先使用 .webp、.png 或 .jpg。
  • 删除图片前先搜索是否仍被页面引用。

6. 构建与检查

每次更新内容或配置后，建议执行生产构建：

npm run build

如果构建通过，可继续预览生产版本：

npm run preview

重点检查：

  • 首页是否能正常打开。
  • 新增或修改的页面是否能访问。
  • 侧边栏分类和页面顺序是否符合预期。
  • 图片、链接、代码块和表格是否正确显示。
  • 控制台或终端是否有构建错误。

7. 发布前核对

发布前建议按照以下顺序处理：

  1. 确认所有内容修改已保存。
  2. 运行 npm run build 并确认无错误。
  3. 必要时运行 npm run preview 检查生产预览。
  4. 确认没有误提交 node_modules/、dist/、.astro/ 等生成目录。
  5. 提交源码变更并推送到远程仓库。
如果项目使用 GitHub Pages 自动部署，通常只需要推送源码到配置好的分支，部署流程会由 GitHub Actions 或 Pages 配置触发。

8. 常见问题

页面没有出现在侧边栏

  • 确认文件位于 src/content/docs/guides/ 或 src/content/docs/reference/。
  • 确认文件扩展名是 .md 或 .mdx。
  • 确认 frontmatter 格式正确，至少包含 title。

构建失败

  • 先阅读终端错误信息，通常会指出具体文件和行号。
  • 检查 frontmatter 的冒号、引号和缩进是否正确。
  • 检查 MDX 中是否存在未闭合的标签或组件。

图片无法显示

  • 检查图片路径相对于当前 Markdown 文件是否正确。
  • 检查图片文件名大小写是否一致。
  • 如果图片放在 public/，引用路径应按站点根路径组织。

9. 维护清单

  • 确认本次修改范围：内容、配置、资源或依赖。
  • 新增页面已包含 title 和 description。
  • 新增图片路径已在页面中验证。
  • 导航和侧边栏显示正常。
  • 外部链接可以打开，内部链接没有 404。
  • npm run build 已通过。
  • 未提交不必要的生成目录或缓存目录。
